# WebToMD v1 MVP — token-efficient webpage clipping for AI agents

## Problem Statement

When feeding web content to an AI agent (chatbot, coding assistant, research tool), users face three friction points:

1. **Manual copy-paste is messy.** Selecting text in a browser produces raw whitespace, broken line breaks, and formatting that the AI must parse through.
2. **Existing web clippers capture too much.** Tools like single-file savers or naive DOM-to-Markdown converters include navigation, sidebars, footers, ads, related-article widgets, and cookie banners — burning AI tokens on noise.
3. **Routing the agent through a browser MCP adds overhead.** When the user already knows which URL is relevant, asking the agent to drive a browser (navigate, scroll, click) burns tokens on navigation overhead and adds latency.

WebToMD v1 solves (1) and (2) with a Manifest V3 Chrome extension. Problem (3) is intentionally deferred to v2.

## Solution

A Chrome extension that, on user action (popup click or right-click → context menu):

1. Extracts the **main content** of the current page using Mozilla Readability.
2. Converts it to Markdown using Turndown.
3. Wraps it in **YAML frontmatter** with metadata: URL, title, site name, author, publication date, and a fetched-at timestamp.
4. Applies one of three named **Presets** that control how images, links, and frontmatter are handled.
5. Writes the resulting Markdown to the clipboard automatically.
6. Surfaces a **live token estimate** in the popup so the user can pick the right Preset before committing.

Two workflows:

- **Page Workflow** — Click the extension icon → popup opens with preset tabs, article preview, and token count → auto-copy on popup open or on preset switch.
- **Selection Workflow** — Right-click selected text → context menu "Copy Selection as MD" → direct Turndown on the selection's HTML → in-page toast with token count.

Architectural commitments (see `docs/adr/`):

- No backend service — all processing client-side. (ADR-0001)
- Built on WXT + Svelte 5 + TypeScript. (ADR-0002)
- Extraction via `@mozilla/readability` + `turndown`. (ADR-0003)
- No host permissions — only `activeTab`. (ADR-0004)
- Three fixed Presets (Chat-ready, Reference, Archive); Chat-ready is default. (ADR-0005)

## User Stories

1. As an AI agent user, I want to copy a webpage's main content as clean Markdown, so that I can paste it into an AI chat without manually cleaning up the text.
2. As an AI agent user, I want the Markdown to exclude navigation, sidebars, footers, ads, and related-article widgets, so that I don't waste tokens on noise.
3. As an AI agent user, I want the Markdown to include the source URL and article title, so that the AI can cite the source in its response.
4. As an AI agent user, I want the Markdown to include publication date and author when available, so that the AI can reason about recency and credibility.
5. As an AI agent user, I want to choose between output formats optimized for different uses (chat context, research with citations, archival), so that I can trade off token cost against information density.
6. As an AI agent user, I want to see an estimate of how many tokens the output will consume, so that I can choose the right format before committing.
7. As an AI agent user, I want to copy a paragraph or code snippet from a page as Markdown, so that I can paste just the relevant piece into an AI chat.
8. As an AI agent user, I want to copy a webpage as Markdown with one click (icon click), so that the workflow is frictionless.
9. As an AI agent user, I want to copy a selection as Markdown via right-click menu, so that I don't have to open the popup for quick extractions.
10. As an AI agent user, I want to see a brief confirmation when a copy succeeds, so that I know the action completed.
11. As a privacy-conscious user, I want the extension to NOT request access to all websites, so that I can trust the install.
12. As a privacy-conscious user, I want all extraction to happen locally on my machine, so that no third party sees the pages I read.
13. As a privacy-conscious user, I want the extension to NOT send any telemetry, so that my usage is private.
14. As a first-time user, I want a welcome page that explains the workflow in 3 steps, so that I can start using the extension immediately.
15. As a return user, I want the welcome page to NOT re-appear on every Chrome update, so that I am not interrupted.
16. As a customizer, I want to change the default Preset in settings, so that the popup pre-selects my preferred format.
17. As a customizer, I want to toggle the frontmatter metadata on/off, so that I can control whether source info is included.
18. As a customizer, I want to revisit the welcome page if I forget how to use the extension.
19. As a developer using a Mac and a Windows PC, I want my settings to sync across devices, so that I have a consistent experience.
20. As an extension user, I want the extension to NOT require me to create an account, so that I can start using it immediately.
21. As an extension user, I want the extension to work on any webpage (not just blogs), so that I can use it on documentation sites, forums, news sites, etc.
22. As an extension user, I want graceful behavior when a page cannot be cleanly extracted (e.g., heavy JavaScript apps), so that I am not stuck with empty output.
23. As an extension user, I want a clear error message when extraction fails, so that I know to try a different approach (e.g., selection).
24. As an extension user, I want the popup to show a preview of the Markdown, so that I can verify the output before pasting.
25. As an extension user, I want the popup to auto-copy on opening, so that I do not need a second click.
26. As an extension user, I want the popup to show the article title, so that I have visual confirmation of what I am copying.
27. As an extension user, I want the popup to be compact (small footprint), so that it does not obscure my page.
28. As a discoverer via the Chrome Web Store, I want the extension's listing to clearly describe what it does, so that I can decide whether to install it.
29. As a developer using the Chrome Web Store, I want the extension to load with minimal permissions, so that the install prompt is not scary.

## Implementation Decisions

**Modules to build** (in `lib/` and `entrypoints/`):

- `lib/extractor/` — wrapper around `@mozilla/readability` for Page Workflow; direct Turndown on selection HTML for Selection Workflow.
- `lib/presets/` — registry of three Presets (Chat-ready / Reference / Archive).
- `lib/frontmatter/` — YAML serializer for Standard metadata fields.
- `lib/token/` — token estimation (chars/4 heuristic for v1).
- `lib/storage/` — typed wrapper around `chrome.storage.sync`.
- `lib/context-menu/` — `chrome.contextMenus` registration and click handler.
- `lib/messaging/` — typed `chrome.runtime.sendMessage` wrappers.
- `lib/clipboard/` — clipboard write with fallback.
- `lib/toast/` — Shadow DOM toast injection.
- `entrypoints/popup/App.svelte` — popup UI (preset tabs, preview, token count, copy).
- `entrypoints/options/Options.svelte` — settings UI.
- `entrypoints/welcome/Welcome.svelte` — first-install welcome.
- `entrypoints/background.ts` — service worker (context menu lifecycle + storage).

**Key interfaces** (all in `types/`):

```ts
// types/clip.ts — already in scaffold
type WebClip = SelectionClip | PageClip;

// types/preset.ts — already in scaffold
interface Preset {
  id: PresetId; // 'chat-ready' | 'reference' | 'archive'
  label: string;
  description: string;
  imagePolicy: ImagePolicy; // 'strip' | 'keep-url'
  linkPolicy: LinkPolicy; // 'strip' | 'keep-inline'
  frontmatter: true;
}
```

**New interfaces (planned)**:

- `formatClip(clip: WebClip, preset: Preset): string` — the pure composition function. This is the "what the user pastes" contract and is the primary test seam.
- `extractPage(tab): Promise<PageClip>` — extraction entry point for Page Workflow.
- `extractSelection(tab): Promise<SelectionClip>` — extraction entry point for Selection Workflow.
- `serializeFrontmatter(meta): string` — YAML frontmatter serializer.
- `estimateTokens(md: string): number` — token estimator.
- `getSettings() / setSettings(s)` — typed storage wrapper.
- `registerContextMenus()` — context menu setup.
- `writeClipboard(text)` — clipboard write with fallback.
- `showToast(tab, message)` — Shadow DOM toast injection.

**Schema contracts**:

- `WebClip` discriminated by `kind: 'selection' | 'page'`.
- Standard frontmatter fields: `url` (required), `title` (required), `fetched_at` (required), `date` (optional), `author` (optional), `site_name` (optional). Fields are omitted (not blanked) when unavailable.
- `Settings = { lastUsedPreset: PresetId, onboardingComplete: boolean, frontmatterEnabled: boolean }`.

**Specific interactions**:

- **Page Workflow**: User clicks icon → popup mounts → popup calls `extractPage(activeTab)` via `chrome.scripting.executeScript()` → popup calls `formatClip(clip, currentPreset)` → popup displays preview + token count → popup auto-writes to clipboard → popup shows success state.
- **Selection Workflow**: User right-clicks selection → context menu "Copy Selection as MD" → background handler calls `extractSelection(activeTab)` → writes formatted Markdown to clipboard → injects toast via `showToast()`.
- **First install**: `chrome.runtime.onInstalled` with `reason === 'install'` opens welcome page in new tab; dismiss sets `onboardingComplete = true`.
- **Settings sync**: All settings via `chrome.storage.sync` so they follow the user across devices.

**Cross-references to ADRs** (in `docs/adr/`):

- ADR-0001: no backend service
- ADR-0002: WXT + Svelte 5 + TypeScript stack
- ADR-0003: `@mozilla/readability` + `turndown`
- ADR-0004: no host permissions; `activeTab` only
- ADR-0005: three Presets, Chat-ready default

## Testing Decisions

**Primary seam**: `formatClip(clip, preset)` — pure function, no Chrome API dependencies. This is the function that defines "what the user pastes" and is the highest-value test surface.

- **Test cases**: All three Presets × both clip kinds × edge cases (missing optional fields, very long content, code blocks, tables, links, images, frontmatter toggled off).
- **What makes a good test**: External behavior only — assert on the resulting Markdown string, not internal intermediate steps.

**Secondary seams** (lower-priority unit tests):

- `lib/frontmatter/` — Vitest snapshot tests for YAML serialization across field combinations.
- `lib/token/` — Vitest unit tests with known input/output.
- `lib/presets/` — assertions on each Preset's policy.
- `lib/extractor/` — Vitest snapshot tests with curated HTML fixtures (e.g., Medium article, GitHub README, Hacker News thread, StackOverflow answer).

**Integration seams** (manual QA only for v1):

- Full Page Workflow on a variety of sites (article, docs, blog, forum).
- Full Selection Workflow on a variety of sites.
- Welcome page on first install vs subsequent updates.
- Cross-device settings sync.

**Prior art**: Vitest is already configured (`vitest.config.ts`); existing `tests/extractor.test.ts` placeholder is the seam.

## Out of Scope (v1)

- URL-to-Markdown service (problem #3 from original brief).
- Smart Clipboard / Append mode (post-MVP, see v2 backlog in `CONTEXT.md`).
- Keyboard shortcut for Selection Workflow.
- Custom user-defined Presets (v1 ships three fixed).
- Per-site Preset overrides.
- Site-specific adapters (Reddit, StackOverflow, Twitter/X, GitHub Discussions, Medium).
- Defuddle swap or alternative extraction library.
- `<meta>` tag parsing for richer frontmatter enrichment.
- Tiktoken-based token estimation (v1 uses chars/4 heuristic).
- Firefox + Safari support (v1 ships Chrome Manifest V3 only).
- Theming / appearance customization.
- i18n / multi-language UI (English only for v1).
- Analytics or telemetry of any kind.
- Paid / freemium model.
